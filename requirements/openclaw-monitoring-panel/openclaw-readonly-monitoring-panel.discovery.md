# OpenClaw Readonly Monitoring Panel Discovery

## Problem Statement

当前仓库虽然已经有 `openclaw-dashboard` 的迁移与后端骨架，但“面向团队运维与非技术同事都能看懂”的监控面板目标仍不够清晰。现状问题主要有：
- 缺少统一视图来实时看到各个 OpenClaw 节点实例的在线状态、健康度和层级关系。
- 缺少按节点查看其下属多个 agent 当前状态、正在处理的任务和最近进展的可读入口。
- 非技术同事无法通过现有技术导向材料快速判断“系统现在在做什么、哪里异常、最近产出了什么”。
- 运维同事缺少秒级、短期历史可追踪的只读面板来判断节点/agent 是否退化、卡住或出现异常波动。

目标是先交付一个只读监控面板，清晰展现 `OpenClaw 节点 -> agents -> 任务时间线` 的所属关系，并同时满足业务摘要和运维明细两类查看需求。

## Goals

- 提供一个只读监控面板，统一展示多个 OpenClaw 节点实例的状态。
- 明确展现每个 OpenClaw 节点下属多个 agent 的层级与归属关系。
- 在首页采用双层结构：
  - 顶部提供非技术同事可读的业务摘要。
  - 下部提供运维可用的技术明细。
- 提供 agent 当前任务的时间线级信息，包括开始时间、持续时间、最近一步进展和当前阶段。
- 提供秒级刷新体验，让节点状态、agent 状态和任务进展能在 1 到 2 秒级别更新。
- 提供短期历史能力，支持查看最近一段时间内的状态变化、任务时间线与异常记录。
- 对非技术同事暴露可读摘要，而不是原始消息正文或工具调用细节。

## Non-Goals

- 第一版不提供任何写操作或控制能力，例如暂停 agent、恢复 agent、重试任务、下线节点。
- 第一版不暴露完整原始会话内容、模型思考、工具调用参数或消息正文片段。
- 第一版不做长期趋势报表，不做按天/按周的运营分析看板。
- 第一版不处理复杂权限分层和细粒度 RBAC，只围绕只读查看设计。
- 第一版不扩展为通用配置管理后台，不承担 token 管理、节点配置修改或运维编排职责。

## Target Users and Scenarios

- 目标用户：
  - 团队运维人员，需要快速判断节点和 agent 是否健康、是否异常、最近是否退化。
  - 非技术同事，需要快速理解系统当前在忙什么、有哪些任务进行中、最近产出了什么摘要。

- 核心场景：
  - 运维人员进入首页，先看整体健康概况，再下钻到某个节点查看其下属 agent 状态与最近异常。
  - 团队成员查看某个 agent 正在处理什么任务、处于哪个阶段、已经跑了多久。
  - 非技术同事进入面板，先看业务摘要，再点开最近任务或最近产出摘要了解当前工作内容。
  - 团队成员查看最近 1 小时或 24 小时内的状态变化与异常记录，用于复盘节点抖动或任务阻塞。

## Constraints
- Technical
  - 监控对象是 OpenClaw 节点实例，每个节点下有多个 agent，需要在 UI 中清晰呈现层级关系。
  - 第一版必须保持只读，不引入节点或 agent 控制操作。
  - 刷新体验目标为 1 到 2 秒级别，不能退化成明显滞后的人工刷新界面。
  - 需要支持短期历史查看，但不扩展为长期统计报表系统。
  - 面板应优先复用仓库现有 `projects/openclaw-dashboard` 方向与已存在的 dashboard 设计资料，而不是另起一套无关架构。
  - 不得把敏感信息、token 或私密配置写入仓库文件。

- Product
  - 首页采用“业务摘要 + 运维明细”的双层结构，而不是纯技术监控界面。
  - “工作内容”在第一版展示为可读摘要与时间线进展，不展示原始会话正文。
  - 节点、agent、任务和异常必须可被非技术同事基本理解，不依赖运维术语才能读懂。

- Timeline
  - 需要先尽快做出 MVP，把只读监控视图跑通。
  - 第一版优先验证信息架构、数据链路和秒级刷新体验，复杂分析能力后置。

## Assumptions

- 假设目录名采用 `requirements/openclaw-monitoring-panel/`，因为用户未单独指定输出目录名，且该命名符合仓库现有 `requirements/<feature>/` 习惯。
- 假设第一版接入的是多个 OpenClaw 节点实例，而不是单节点单 agent 的极简场景。
- 假设现有 OpenClaw 侧或 dashboard backend 能提供节点状态、agent 状态、任务阶段、最近进展与短期历史所需的基础数据。
- 假设短期历史的产品口径更接近“最近 1 小时或 24 小时”，最终时长可在技术方案阶段再细化。
- 假设“最近产出”适合先展示标题、状态和摘要，而不包含原始内容全文。
- 假设第一版主要面向内部团队使用，因此可在视觉与字段命名上兼顾技术和非技术用户，但无需先引入复杂权限模型。

## Acceptance Criteria

- 信息架构
  - 首页必须同时包含业务摘要区和运维明细区。
  - 页面必须清楚呈现 `OpenClaw 节点 -> agents` 的所属关系。

- 只读范围
  - 第一版界面中不出现暂停、恢复、重试、下线、修改配置等写操作入口。

- 节点与 agent 可观测性
  - 每个节点至少展示在线状态、最近活跃时间、异常状态或健康状态。
  - 每个 agent 至少展示所属节点、当前状态、当前任务、开始时间、持续时间、最近一步进展和当前阶段。

- 工作内容可读性
  - 非技术同事能够在不阅读原始会话内容的前提下，看懂“当前在做什么”和“最近有什么产出/进展”。
  - 最近任务或最近产出至少提供标题、状态和摘要。

- 刷新体验
  - 节点状态、agent 状态和任务进展能够以 1 到 2 秒级别更新。

- 短期历史
  - 系统能够查看最近一段时间内的状态变化、任务时间线或异常记录。
  - 至少支持一个明确的短期历史窗口，候选为最近 1 小时或最近 24 小时。

- 目标用户适配
  - 运维人员可以从同一面板中快速发现异常节点、异常 agent 或停滞任务。
  - 非技术同事可以从同一面板中快速理解系统当前工作概况，而不是只看到底层技术指标。

## Open Questions

- 短期历史窗口最终定为最近 1 小时、最近 24 小时，还是二者都支持。
- 业务摘要区中的“最近产出”究竟展示任务产物摘要、最近完成任务摘要，还是两者合并。
- 节点与 agent 的异常定义口径是什么，例如断连、长时间无进展、失败率升高、心跳延迟等是否都算异常。
- 第一版是否需要区分不同 agent 类型、不同业务来源或不同接入渠道。
- 非技术同事是否需要固定的简化视图，还是同一页面通过信息层次自然兼容即可。
